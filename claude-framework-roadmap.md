## 1. Зачем это всё (главный вывод)

Базовая настройка Claude под аналитика и Architecture Framework — это **инструмент** и **методология**. Сами по себе они не сцеплены: Claude не знает про шаблоны, ID, трассировку и валидацию фреймворка. Цель этой дорожной карты — **сцепить их** так, чтобы Claude:

1. генерировал артефакты **строго по шаблонам** (однотипность, читаемость);
2. вёл **сквозной реестр требований** и базу знаний проекта (second brain → RAG);
3. гарантировал **отсутствие деградации** через трассировку и валидацию;
4. начинал **любой проект с дорожной карты** как единой точки входа.

---

## 2. Принципы

- **Единый источник правды — git-репозиторий**, проверяемый в CI (а не Confluence/переписки).
- **Жизненный цикл артефакта:** `Draft → Approved → (RAG — позже)`. Статус хранится **явным полем в документе** и дублируется в **отдельном реестре approved**. В векторную базу попадёт только approved — но RAG отложён (сначала second brain).
- **Однотипность через шаблоны:** документы создаются только скиллами-генераторами, а не «от руки».
- **Целостность через трассировку + валидацию:** каждый артефакт имеет ID и проверяемые связи; `validate_docs.py` блокирует merge при нарушениях.
- **Дорожная карта — вход в проект:** старт любого проекта = прохождение 6 этапов `ROADMAP_design.md`.

---

## 3. Жизненный цикл артефакта (ключевая механика)

```
[Draft] ──ревью + /review-doc + validate_docs.py──> [Approved] ──(позже)──> [RAG ⏸]
   │                                                     │
 в репозитории (second brain):                  статус = явное поле          отложено:
 доступно Claude и команде,                     в документе + запись         векторизация
 включая черновики                              в реестре approved           approved по документу
```

- **Second brain (делаем сейчас):** все артефакты (включая Draft) живут в репозитории и реестре — Claude и команда видят их и работают с ними. Это активная цель.
- **Approval — явный шаг:** перевод в `Approved` делает ревьюер, выставляя **поле статуса в документе** и добавляя запись в **отдельный реестр approved**. Не привязано к merge: документ может быть слит в репозиторий, оставаясь Draft.
- **RAG (⏸ отложено):** когда накопятся approved-артефакты — они векторизуются (по документу) и попадают в проектную базу; Claude опирается на них при генерации. Это защита от деградации: новые артефакты опираются на утверждённые, а не «додумываются». Стек выбираем на Фазе 2. **План отката:** если RAG начнёт галлюцинировать на approved — векторизация отключается, Claude возвращается к second brain (реестр + файлы), без потери данных.

## 4. Рабочие потоки

Легенда статусов: ☐ не начато · ◐ в работе · ☑ готово · ⊘ заблокировано
Приоритеты: **P0** критично · **P1** важно · **P2** желательно

### WS1 — Скиллы-генераторы документов (P0)

**Цель:** все артефакты создаются командами, строго по шаблонам фреймворка, с обязательным разделом Traceability.

| # | Задача | Приоритет | Статус |
|---|---|---|---|
| 1.1 | Синхронизировать `/adr` под шаблон фреймворка (`docs/architecture/adr/_template_adr.md`) и путь | P0 | ☐ |
| 1.2 | Скилл `/feature` — Feature Spec по шаблону (11 обязательных разделов + ID) | P0 | ☐ |
| 1.3 | Скилл `/slices` — Behavior Slices (ядро фреймворка) | P0 | ☐ |
| 1.4 | Скилл `/state-machine` — Screen State Machine | P1 | ☐ |
| 1.5 | Скиллы контрактов: `/api-contract`, `/errors`, `/golden-sample` | P2 | ☐ |
| 1.6 | Каждый скилл проставляет ID и заполняет Traceability автоматически | P0 | ☐ |

**Критерии готовности:** сгенерированный документ проходит `validate_docs.py` без правок; формат идентичен шаблону.
**Размещение:** скиллы — проектные, в `.claude/skills/` репозитория (общие для команды и Сергея); личное — в `settings.local.json`.

### WS2 — Сквозной реестр требований + RAG (P0 → P1)

**Цель:** единая база знаний проекта; защита от деградации через опору на утверждённые артефакты.

| # | Задача | Приоритет | Статус |
|---|---|---|---|
| 2.1 | Автоведение `Artifact Registry` (`docs/_meta/artifacts.md`): скиллы из WS1 дописывают реестр при создании артефакта | P0 | ☐ |
| 2.2 | Явное поле статуса `Draft/Approved` в каждом артефакте | P0 | ☐ |
| 2.3 | Реестр approved = колонки `Status` / `approved_date` / `reviewed_by` в `artifacts.md` (единый источник); поле в документе — человекочитаемая копия | P0 | ☐ |
| 2.4 | «Second brain» = навигация по реестру + указатели в `CLAUDE.md` (Claude уже читает файлы через grep/glob — отдельный поисковый движок не нужен) | P1 | ☐ |
| 2.5 | RAG (⏸ отложено): векторизация approved по документу | P2 | ⏸ |
| 2.6 | RAG (⏸ отложено): извлечение из базы при генерации (опора вместо догадок) | P2 | ⏸ |
| 2.7 | RAG (⏸ отложено): выбор стека (эмбеддинги, хранилище, где исполняется) | P2 | ⏸ |

**Критерии готовности (second brain):** реестр всегда консистентен; у каждого артефакта есть статус; approved-артефакты собраны в отдельном реестре; деградации связей нет.

## 6.1 Метрики успеха

Как поймём, что сцепление работает (а не превращается в бюрократию):
- **Время «запрос → готовый Draft»** — снижается.
- **Доля документов, проходящих валидацию с первой попытки** — растёт.
- **Частота противоречий между артефактами** (ловится трассировкой) — снижается.

## 7. Принятые решения и оставшиеся вопросы

**Решено (2026-06-08):**
- **Approval:** статус `Draft/Approved` — **явное поле в документе**; источник правды — **колонки `Status` / `approved_date` / `reviewed_by` в `artifacts.md`**, поле в документе — копия. Валидатор сверяет их (WS3.5). Не привязано к merge PR (документ может быть слит, оставаясь Draft).
- **Роль ревьюера:** `Approved` ставит **автор документа после подтверждения Сергея** (или назначенного лица); в реестре фиксируется `reviewed_by`. Валидатор проверяет наличие поля, но не подлинность подтверждения — это организационная дисциплина.
- **Ввод валидации в CI — поэтапно:** сначала режим **warning** (не блокирует), после отладки генераторов — **блокирующий** (WS3.3 → WS3.4).
- **RAG:** ⏸ **отложен**. Сначала second brain; RAG включаем, когда накопятся approved-артефакты. Индексация — **по документу**. Есть план отката (раздел 3).
- **Скиллы:** проектные, коммитим в `.claude/skills/` репозитория; личное — в `settings.local.json`.

**Осталось решить (не блокирует старт):**
- Конкретный стек RAG (эмбеддинги, хранилище) — вернуться на Фазе 2.
