# Настройка Claude Code под аналитическую документацию и ADR

Полный гайд: от глобального конфига (`~/.claude/`) до конфига отдельного проекта (`.claude/`),
плюс готовые к копированию скиллы, сабагент и output-style.

Ориентировано на работу с прозой (аналитика, ADR, требования), а не код. Презентации и Excel —
отдельным блоком в конце.

---

## 0. Карта: что есть что и куда кладётся

| Механизм | Глобально (все проекты) | В проекте | Зачем тебе |
|---|---|---|---|
| **CLAUDE.md** (память) | `~/.claude/CLAUDE.md` | `./CLAUDE.md` | Всегда-истинные правила: язык, стиль, шаблоны, терминология |
| **settings.json** | `~/.claude/settings.json` | `.claude/settings.json` + `.claude/settings.local.json` | Модель, output-style, права, env |
| **output-styles** | `~/.claude/output-styles/*.md` | `.claude/output-styles/*.md` | Заменяет «кодовый» системный промпт на аналитический |
| **skills** (твои команды) | `~/.claude/skills/<имя>/SKILL.md` | `.claude/skills/<имя>/SKILL.md` | `/adr`, `/refine`, `/to-confluence`, `/review-doc` |
| **agents** (сабагенты) | `~/.claude/agents/<имя>.md` | `.claude/agents/<имя>.md` | Исследование исходников в изолированном контексте |
| **plugins** | через `/plugin` | — | superpowers, elements-of-style |

**Правило приоритета:** более конкретное перекрывает более общее. Проектный CLAUDE.md
побеждает пользовательский; проектный скилл/сабагент с тем же именем побеждает глобальный.

**Логика выбора инструмента:**
- Всегда верное короткое правило → **CLAUDE.md**
- «Вставь этот шаблон промпта по команде» → **skill** с `disable-model-invocation: true`
- «Применяй сам, когда задача подходит» → **skill** с описанием (без `disable-model-invocation`)
- «Изолировать контекст, чтобы не засорять основной тред» → **сабагент**
- «Сменить тон/регистр всех ответов» → **output-style**

## 1. Уровень 0 — плагины

Один раз, глобально. Внутри Claude Code:

```
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
/plugin install elements-of-style@superpowers-marketplace
```

Что берём:
- **superpowers** → команды `/superpowers:brainstorm`, `/write-plan`, `/execute-plan` и автоскиллы.
  Для тебя ценны brainstorm (сократический разбор требований) и планирование больших документов.
  TDD/дебаг-часть на прозе просто не активируется — не мешает.
- **elements-of-style** → скилл `writing-clearly-and-concisely` (Strunk). Твой `/review-doc`
  будет на него опираться.

Проверить, что встало: `/help` (появятся команды) и `/plugin`.

> Если в твоей корпоративной среде маркетплейс obra недоступен, superpowers есть и в официальном:
> `/plugin install superpowers@claude-plugins-official`.

## 2. Уровень 1 — глобальный конфиг (`~/.claude/`)

Это твоя личная база, общая для всех проектов.

### 2.1. `~/.claude/settings.json`

```json
{
  "model": "claude-opus-4-8",
  "outputStyle": "analyst",
  "env": {
    "CLAUDE_CODE_SUBAGENT_MODEL": "claude-sonnet-4-6"
  }
}
```

- `model` — дефолтная модель сессии. Тяжёлый анализ держи на Opus; рутину переключай на лету `/model`.
- `outputStyle` — имя кастомного стиля из п. 2.3 (файл `analyst.md` → имя `analyst`).
- `CLAUDE_CODE_SUBAGENT_MODEL` — на чём по умолчанию крутятся сабагенты (исследование можно гонять
  на модели полегче/подешевле).

### 2.2. `~/.claude/CLAUDE.md` — твои всегда-истинные правила

Держи коротким (цель — до ~200 строк, иначе раздувает контекст и хуже соблюдается).

```markdown
# Личные правила (Юрий, системный аналитик)

## Кто я и что делаю
- Я системный аналитик. Пишу аналитическую документацию, ADR и требования. Код — редко.
- Не предлагай писать тесты/рефакторинг/CI, если я явно не прошу про код.

## Язык и стиль
- Всегда отвечай и пиши документы на русском.
- Стиль: ясно, по делу, без воды и канцелярита. Короткие абзацы.
- Сначала вывод/решение, потом обоснование. Списки — только где они реально нужны.
- Не используй эмодзи в документах.

## Как работать со мной
- Если требований не хватает — задай уточняющие вопросы до написания, а не угадывай.
- Перед большим документом покажи структуру (оглавление) и дай согласовать.
- Технические термины и аббревиатуры домена сохраняй как есть, не «переводи».
```

Команды: `/init` — сгенерировать каркас, `/memory` — открыть на редактирование,
`# текст` прямо в диалоге — быстро дописать правило.

### 2.3. `~/.claude/output-styles/analyst.md` — аналитический регистр

Output-style **заменяет** «софт-инженерную» часть системного промпта Claude Code на твою.
Все инструменты (чтение/запись файлов, скрипты, MCP) остаются доступны — отключается только
кодоцентричное поведение.

```markdown
---
name: analyst
description: Регистр системного аналитика — проза, документация, ADR и требования вместо кода.
---

# Роль

Ты помогаешь системному аналитику готовить документацию, ADR и требования.
Твой основной продукт — текст для людей, а не код.

## Как ты пишешь
- На русском, ясно и сжато. Главный вывод — в начале.
- Прозой. Списки и таблицы — только когда они объективно понятнее текста.
- Не добавляй код-комментарии, диффы и «давай напишем тест», если об этом не просят.
- Явно отделяй факты от допущений. Допущения помечай.

## Как ты рассуждаешь
- Перед длинным документом предлагаешь структуру и ждёшь согласования.
- В спорных решениях показываешь альтернативы и их компромиссы (как в ADR).
- Если данных не хватает — спрашиваешь, а не выдумываешь.
```

> Хочешь сохранить инженерную дисциплину для редких скриптов — добавь в фронтматтер
> `keep-coding-instructions: true`.

Активация: открой `/config` и выбери стиль, либо просто пропиши `"outputStyle": "analyst"`
в settings.json (см. 2.1). Изменение применяется после `/clear` или нового запуска сессии.
Отдельной команды `/output-style` больше нет — её убрали, управляй через `/config` или настройку.

### 2.4. Глобальные скиллы

Кладём в `~/.claude/skills/` те команды, что нужны в любом проекте: `/adr` и `/review-doc`
(полные тексты — в разделе 4). Имя папки = имя команды.

```
~/.claude/skills/
├── adr/
│   └── SKILL.md          → команда /adr
└── review-doc/
    └── SKILL.md          → команда /review-doc
```

### 2.5. Глобальный сабагент-исследователь

`~/.claude/agents/doc-researcher.md` — читает кучу исходников в своём контексте и возвращает
выжимку, не засоряя основной тред (полный текст — раздел 5).

## 3. Уровень 2 — конфиг под проект (`.claude/`)

Заводится в корне каждого рабочего репозитория/папки. То, что командное и в гит, — в
`CLAUDE.md` и `.claude/settings.json`; личное — в `.claude/settings.local.json`.

### 3.1. `./CLAUDE.md` — память проекта

```markdown
# Проект: <название>

## Контекст
- Что это за система, для кого, какие ключевые компоненты.
- Стейкхолдеры и где лежит источник правды (Confluence space, репозиторий схем и т.д.).

## Терминология и аббревиатуры
- <Термин> — <расшифровка/значение в этом проекте>
- ...

## Конвенции документации
- ADR храним в `docs/adr/NNNN-краткое-название.md`, нумерация сквозная.
- Диаграммы — PlantUML в `docs/diagrams/`.
- Итоговые материалы уезжают в Confluence space "<KEY>", формат — см. скилл /to-confluence.

## Куда смотреть
@docs/glossary.md
@docs/adr/0001-template.md
```

`@path` подтягивает внешние файлы в контекст (глоссарий, шаблон ADR). Пути относительные или
абсолютные, вложенность до 5 уровней. Учти: импорт не уменьшает расход контекста — содержимое
всё равно грузится при старте. Поэтому импортируй только то, что реально нужно всегда.

### 3.2. `.claude/settings.json` (в гит) и `.claude/settings.local.json` (личное, в `.gitignore`)

Командный `.claude/settings.json`:

```json
{
  "outputStyle": "analyst"
}
```

Личный `.claude/settings.local.json` (не коммитить):

```json
{
  "model": "claude-opus-4-8"
}
```

В `.gitignore` проекта:

```
.claude/settings.local.json
```

### 3.3. Скиллы проекта

В `.claude/skills/` кладём то, что специфично для проекта:
- `/to-confluence` — конвертер под формат именно вашего Confluence (раздел 4.3).
- `/refine` — сократический разбор требований, завязанный на терминологию проекта (4.2).

```
.claude/skills/
├── to-confluence/
│   └── SKILL.md
└── refine/
    └── SKILL.md
```

### 3.4. Конфиг под подпапки (для монорепо/больших проектов)

Можно положить вложенный `CLAUDE.md` в подкаталог — он подгрузится, когда Claude работает с
файлами в этой папке. Удобно, если, например, `docs/adr/` требует одних правил, а
`docs/requirements/` — других. Родительские CLAUDE.md держи лёгкими: они грузятся каждую сессию.

```
project/
├── CLAUDE.md                  ← общий для проекта
├── docs/
│   ├── adr/
│   │   └── CLAUDE.md          ← правила именно для ADR
│   └── requirements/
│       └── CLAUDE.md          ← правила для требований
```

## 4. Готовые скиллы (копировать целиком)

Поля фронтматтера: `description` — главное (по нему Claude решает, релевантен ли скилл, пиши в
третьем лице и конкретно). `disable-model-invocation: true` — скилл срабатывает **только** по
твоей команде `/имя`, сам не активируется. `allowed-tools` — какие инструменты разрешены без
переспроса. `argument-hint` — подсказка по аргументам. Аргумент доступен в теле как `$ARGUMENTS`.

### 4.1. `~/.claude/skills/adr/SKILL.md` → `/adr`

```markdown
---
name: adr
description: Создаёт Architecture Decision Record по стандартному шаблону. Используется, когда пользователь вызывает /adr или просит оформить архитектурное решение.
disable-model-invocation: true
argument-hint: [краткое-название-решения]
allowed-tools: Read, Write, Edit, Grep, Glob
---

# Создание ADR

Тема решения передана в аргументе: $ARGUMENTS

## Шаги
1. Если контекст решения неясен — задай 2–4 уточняющих вопроса и дождись ответов.
2. Определи следующий свободный номер в `docs/adr/` (формат `NNNN-краткое-название.md`).
3. Сгенерируй файл строго по шаблону ниже. Пиши на русском, по делу.
4. Покажи путь созданного файла.

## Шаблон

# ADR-NNNN: <Название решения>

- **Статус:** Предложено | Принято | Отклонено | Заменено ADR-XXXX
- **Дата:** <YYYY-MM-DD>
- **Авторы:** <кто>

## Контекст
<Какая проблема и какие силы действуют. Факты и ограничения. Допущения помечай явно.>

## Рассмотренные варианты
1. <Вариант A> — плюсы / минусы / компромиссы
2. <Вариант B> — плюсы / минусы / компромиссы

## Решение
<Что выбрали и почему именно это.>

## Последствия
- Положительные:
- Отрицательные / риски:
- Что теперь нужно сделать:
```

> ⚠️ **Согласовать с Architecture Framework.** Этот шаблон ADR — упрощённый. Во фреймворке ADR требует поля `Artifact ID` и раздела `Traceability` (см. `Architecture Framework/docs/architecture/adr/_template_adr.md`). При синхронизации `/adr` (задача WS1.1 дорожной карты `claude-framework-roadmap.md`) шаблон нужно привести к фреймворк-совместимому и хранить ADR в `docs/architecture/adr/`.

### 4.2. `.claude/skills/refine/SKILL.md` → `/refine`

```markdown
---
name: refine
description: Сократический разбор требований до начала написания документа — вытягивает контекст, границы и критерии приёмки. Используется по команде /refine.
disable-model-invocation: true
argument-hint: [тема-или-фича]
allowed-tools: Read, Grep, Glob
---

# Уточнение требований

Тема: $ARGUMENTS

Не пиши документ сразу. Сначала проведи короткий разбор:

1. Задавай по одному вопросу за раз, не вываливай всё списком.
2. Покрой: цель и проблему, стейкхолдеров, границы (что в скоупе/вне), допущения,
   ограничения (нефункциональные), критерии приёмки, открытые вопросы.
3. Опирайся на терминологию из CLAUDE.md проекта — не вводи синонимы.
4. Когда картина полная — собери краткое резюме требований и предложи,
   во что это оформить (требование / ADR / аналитическая записка).
```

> Можно совместить с `/superpowers:brainstorm` для более глубокого разбора и со Spec Kit,
> если используешь его под requirements.

### 4.3. `.claude/skills/to-confluence/SKILL.md` → `/to-confluence`

```markdown
---
name: to-confluence
description: Конвертирует готовый markdown-документ в формат, который чисто вставляется в Confluence. Используется по команде /to-confluence.
disable-model-invocation: true
argument-hint: [путь-к-файлу]
allowed-tools: Read, Write
---

# Перенос в Confluence

Источник: $ARGUMENTS (если путь не указан — спроси, какой документ конвертировать).

## Что сделать
1. Прочитай markdown-документ.
2. Сконвертируй в Confluence wiki markup:
   - Заголовки `#`/`##`/`###` → `h1.`/`h2.`/`h3.`
   - Списки `-` → `*`, нумерованные → `#`
   - Код-блоки → `{code}` ... `{code}`; инлайн-код → `{{...}}`
   - Таблицы → `||заголовок||...||` и `|ячейка|...|`
   - Ссылки `[текст](url)` → `[текст|url]`
3. Сохрани результат рядом с исходником, суффикс `.confluence.txt`.
4. Кратко напомни: в редакторе Confluence вставлять через «Insert → Markup → Wiki markup».

## Альтернатива
Если в этом space включена вставка Markdown напрямую — отдельно выведи «очищенный» markdown
(без HTML-вставок и кастомных расширений), который Confluence корректно распознаёт при paste.
```

> Это под перенос из изолированной сети вручную. Если до Confluence есть сетевой доступ —
> можно вместо этого подключить Confluence через MCP и публиковать напрямую (см. раздел 7).

### 4.4. `~/.claude/skills/review-doc/SKILL.md` → `/review-doc`

```markdown
---
name: review-doc
description: Проверяет документ по чек-листу качества (ясность, полнота, консистентность) и применяет принципы Strunk. Используется по команде /review-doc.
disable-model-invocation: true
argument-hint: [путь-к-файлу]
allowed-tools: Read, Edit, Grep
---

# Ревью документа

Документ: $ARGUMENTS

Прогони по чек-листу и выдай замечания, сгруппированные по приоритету
(критично / стоит исправить / на подумать):

## Ясность и стиль
- Длинные предложения, канцелярит, вода, пассив там, где уместен актив.
- Применяй принципы writing-clearly-and-concisely (Strunk), если плагин установлен.

## Полнота
- Есть ли контекст, границы, допущения, критерии приёмки/последствия (для ADR).
- Нет ли висящих ссылок на несуществующие разделы/документы.

## Консистентность
- Единая терминология (сверь с глоссарием/CLAUDE.md проекта).
- Единый формат заголовков, нумерации, таблиц.

В конце — короткий вердикт: готов к публикации / нужны правки.
Правки в текст вноси только если я прямо попрошу.
```

## 5. Готовый сабагент

`~/.claude/agents/doc-researcher.md` — для задач, где надо прочитать много исходников
(старые ADR, спеки, переписки, выгрузки) и вернуть выжимку. Работает в отдельном контексте,
основной тред остаётся чистым.

```markdown
---
name: doc-researcher
description: Исследует и собирает выжимку из множества исходных материалов (документы, спеки, ADR) для последующего анализа. Возвращает структурированное резюме, а не сырьё. Используется для ресёрча перед написанием документа.
tools: Read, Grep, Glob, WebFetch, WebSearch
model: inherit
---

Ты — ресёрч-ассистент системного аналитика. Тебе дают набор материалов и вопрос.

Твоя задача:
1. Прочитать указанные источники (файлы, ссылки).
2. Извлечь только релевантное вопросу.
3. Вернуть структурированную выжимку: ключевые факты, противоречия между источниками,
   пробелы (чего не хватает), список источников.

Не пиши финальный документ — твой продукт это выжимка для основного агента.
Чётко отделяй факты от интерпретаций. Если источники противоречат — покажи это явно.
```

> `model: inherit` — использовать модель основной сессии; либо убери строку, чтобы работала
> дефолтная сабагентная модель из `CLAUDE_CODE_SUBAGENT_MODEL`.

Создавать сабагентов можно и интерактивно — командой `/agents`.

## 6. Как это работает вместе — типовой цикл

Допустим, готовишь ADR по выбору механизма транспорта между сервисами.

```
1. /refine транспорт между сервисами
   → Claude задаёт уточняющие вопросы, собирает требования и границы.

2. (опционально) делегируешь ресёрч:
   "Используй сабагент doc-researcher: прочитай docs/adr/ и собери,
    какие транспорты мы уже выбирали и почему."
   → выжимка приходит в основной тред, не раздувая его.

3. /superpowers:brainstorm
   → углублённый разбор альтернатив, если решение нетривиальное.

4. /adr выбор транспорта между сервисами
   → генерится ADR-NNNN по твоему шаблону.

5. /review-doc docs/adr/0042-vybor-transporta.md
   → чек-лист качества + Strunk, замечания по приоритету.

6. /to-confluence docs/adr/0042-vybor-transporta.md
   → готовый wiki markup для вставки в Confluence.
```

Output-style `analyst` и оба CLAUDE.md работают фоном весь цикл — язык, тон и терминология
выдерживаются без напоминаний.

---

## 7. Презентации, Excel и Confluence через MCP

- **Презентации и таблицы.** Claude Code умеет генерить `.pptx`/`.xlsx` через исполнение кода,
  но если это регулярная часть работы — посмотри отдельные инструменты под слайды и таблицы
  (Claude for PowerPoint / Claude for Excel): там работа идёт прямо в документе, а не через CLI.
- **Confluence через MCP.** Если из рабочей среды есть сетевой доступ к Confluence, можно
  подключить его MCP-сервером (`/mcp`) и публиковать страницы напрямую — тогда `/to-confluence`
  не нужен. В изолированной сети остаётся ручной перенос через wiki markup.
